Dos servicios que trabajan juntos

🏗️

Microservicios

El sistema se divide en dos servicios independientes que se comunican por HTTP. Cada uno con una responsabilidad clara.

🖼️

Subida segura

El usuario se registra, hace login y recibe un token. Con ese token sube imágenes. Sin token válido, el servidor lo rechaza.

🔵🐍

Go + Python

Python (FastAPI) maneja usuarios y autenticación. Go (Gin) maneja recepción y almacenamiento de imágenes.

🐳

Docker Compose

Ambos servicios corren en contenedores aislados. Un solo comando levanta todo sin instalar Go ni Python.

Qué hace y qué no hace

Acotar el alcance es parte del diseño: dejar claro qué queda fuera muestra decisiones deliberadas.

SÍ HACE

🔐 Autenticación

Registro con contraseña hasheada, login con JWT firmado y verificación del token en cada petición protegida.

SÍ HACE

📤 Ingesta de imágenes

Valida tipo y tamaño, genera nombre único y guarda en disco solo si el token es válido.

SÍ HACE

🤝 Comunicación

Go consulta a Python en cada upload. Una sola fuente de verdad sobre la validez de los tokens.

NO HACE

🖼️ Procesar imágenes

No redimensiona, comprime ni aplica filtros. Solo ingesta y almacenamiento.

NO HACE

🌐 Galería pública

No sirve las imágenes de vuelta ni expone una galería navegable.

NO HACE

👥 Roles finos

No hay administradores ni niveles. Todo usuario autenticado tiene los mismos permisos. Base extensible.

import bcrypt, os PEPPER = os.getenv("PEPPER").encode() # secreto fuera de la BD def hash_password(password: str) -> str: mezcla = password.encode() + PEPPER # pepper + salt + cost lento return bcrypt.hashpw(mezcla, bcrypt.gensalt(rounds=12)).decode() def verify_password(password: str, hashed: str) -> bool: return bcrypt.checkpw(password.encode() + PEPPER, hashed.encode())

@app.post("/login") def login(datos: LoginInput): user = get_user(datos.username) if not user or not verify_password(datos.password, user.hashed): raise HTTPException(401, "credenciales inválidas") payload = {"sub": user.username, "exp": datetime.utcnow() + timedelta(minutes=30)} # expira 30 min return {"access_token": jwt.encode(payload, SECRET, algorithm="HS256")} @app.get("/api/auth/verify") def verify(authorization: str = Header(None)): try: jwt.decode(authorization.replace("Bearer ", ""), SECRET, algorithms=["HS256"]) return {"valid": True} except jwt.PyJWTError: raise HTTPException(401, "token inválido o expirado")
core/auth.py — cifrado AES-CBC_□×
from Crypto.Cipher import AES from Crypto.Util.Padding import pad, unpad import os, base64 KEY = os.getenv("AES_KEY").encode() # 32 bytes = AES-256 def encrypt(texto: str) -> str: iv = os.urandom(16) # IV aleatorio por cada cifrado cipher = AES.new(KEY, AES.MODE_CBC, iv) ct = cipher.encrypt(pad(texto.encode(), AES.block_size)) # padding return base64.b64encode(iv + ct).decode() # IV viaja con el cifrado def decrypt(token: str) -> str: raw = base64.b64decode(token) iv, ct = raw[:16], raw[16:] # separa IV del cifrado cipher = AES.new(KEY, AES.MODE_CBC, iv) return unpad(cipher.decrypt(ct), AES.block_size).decode()
🔐Ciberseguridad
_□×

Capas de seguridad implementadas

🔐

bcrypt + pepper

Contraseñas hasheadas con bcrypt más un secreto pepper. Aunque roben la BD, no recuperan las contraseñas.

🎫

JWT + expiración

Token JSON firmado que expira en 30 min. Sin token válido: 401 Unauthorized.

🤝

Verificación entre servicios

Go no verifica solo: pregunta a Python en cada request vía /api/auth/verify.

🔒

AES-CBC

Cifrado simétrico de datos con IV aleatorio por operación y padding de bloque. Sin patrones repetidos.

📁

Validación de archivos

Whitelist: solo jpg, png, gif, webp. Máximo 10 MB. Los .exe/.php se rechazan antes del disco.

🔑

Variables de entorno

JWT_SECRET, PEPPER y AES_KEY se inyectan por entorno en docker-compose, nunca hardcodeados.

DEFENSA EN CAPAS Ninguna capa sola es perfecta. Apiladas — hash, JWT, verificación entre servicios, validación, cifrado y CORS — hacen el sistema difícil de romper. Mismo principio que el Zero Trust de la red.
🌐Infraestructura de Red
_□×

La red física detrás del proyecto

Desplegado sobre una red real: 5 VLANs, cuatro switches con EtherChannel y una Raspberry Pi como router. Cada decisión tiene una razón técnica.

🗺️ Segmentación con VLANs + VLSM

VLANGrupoSubredGatewayHostsServidor
10Microservicios192.168.0.128/27.12930auth-service + image-service
20Streaming192.168.0.64/26.6562Streaming Server — VLC
30Mail192.168.0.0/26.162Mail Server — SMTP + SQLite
40Gaming192.168.0.160/27.16130Counter Strike Server
50Invitados192.168.0.192/27.19330WiFi visitantes — solo internet
¿POR QUÉ? Sin VLANs, un equipo en gaming podría atacar al mail server. Las VLANs crean fronteras lógicas: aunque compartan switch físico, el tráfico está aislado. 802.1Q etiqueta cada frame con su VLAN.

⚡ Topología EtherChannel — 4 switches

SW0 — Core
Fa0/3–Fa0/4 → SW1 · Fa0/5–Fa0/6 → SW2
│ PAgP desirable │
SW1 — Access
→ SW0 (PAgP) · → SW3 (LACP)
│ LACP active │
│ PAgP desirable │
SW2 — Access
→ SW0 (PAgP) · → SW3 (ON)
│ ON estático │
SW3 — Access
→ SW1 (LACP) · → SW2 (ON)
🟣

PAgP

Propietario de Cisco, modo desirable. Solo entre switches Cisco → enlaces del core.

🔵

LACP

Estándar IEEE 802.3ad, multi-vendor. Modo active. Más robusto para entornos mixtos.

🟠

ON

Sin negociación: ambos lados fuerzan el canal. Simple, pero sin detección de errores.

🍓 Router-on-a-Stick con Raspberry Pi

🍓 Raspberry Pi
eth0 — trunk 802.1Q
⬇ un solo cable físico ⬇
eth0.10 .129 eth0.20 .65 eth0.30 .1 eth0.40 .161 eth0.50 .193
⬇
SW0 Core Switch
puerto trunk → Pi

En vez de una interfaz física por VLAN (5 puertos), usamos un solo puerto físico y 5 subinterfaces lógicas por software. Eficiente y escalable.

Cuando un equipo de la VLAN 20 quiere hablar con la VLAN 10, el frame sube por el trunk a la Pi, la Pi enruta, y baja a la VLAN destino — todo por el mismo cable.

La Pi también hace NAT masquerade: traduce las IPs privadas a su IP pública para dar salida a internet.

🔒 Zero Trust — Firewall entre VLANs

Por defecto, todo el tráfico inter-VLAN está bloqueado. Solo cruzan los flujos permitidos explícitamente.

✅
VLAN 20 → 10

Streaming accede al auth-service (:8081) para autenticarse. Solo ese puerto.

✅
VLAN 30 → 10

Mail consulta el auth-service para verificar usuarios. SMTP queda en VLAN 30.

🚫
VLAN 40 → todo

El CS Server está aislado. No habla con otros servidores, solo WAN.

🚫
VLAN 50 → servidores

Invitados tienen internet (NAT) pero cero acceso a servidores.

ZERO TRUST Si comprometen el CS Server o un invitado, no hay movimiento lateral hacia el mail server ni los microservicios. Cada segmento confía en cero — igual que el middleware que verifica el JWT en cada request.

De cero a funcionando en 5 pasos

1

Requisitos

Solo Docker Desktop y Git. Docker construye todo; no necesitas Go ni Python en tu máquina.

# verifica docker docker --version
2

Clona el repositorio

git clone https://github.com/Piwody/Integrador-.git cd Integrador-
3

Levanta los servicios

Un comando construye Go y Python y los inicia.

docker-compose up --build ✔ auth-service Started · image-service Started
4

Sirve el frontend

En una segunda terminal, para evitar bloqueos de CORS.

python -m http.server 3000 http://localhost:3000/index.html
5

Usa el sistema

Regístrate, inicia sesión y sube tu imagen. Para detener: docker-compose down
